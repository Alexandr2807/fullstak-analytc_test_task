# Task 3.2 — Алгоритм создания пользователя на backend

Ниже приведен подробный пошаговый алгоритм обработки запроса на регистрацию пользователя после нажатия кнопки **Register** на frontend.

---

## Пошаговый алгоритм

1. Backend получает HTTP POST запрос на `/api/v1/users/register`.

2. Backend проверяет наличие тела запроса.
   - Если тело отсутствует или передан некорректный JSON, вернуть ошибку `400 Bad Request`.

3. Backend извлекает из запроса поля:
   - `firstName`
   - `lastName`
   - `email`
   - `password`
   - `confirmPassword`

4. Backend проверяет наличие обязательных полей.
   - Если хотя бы одно обязательное поле отсутствует или пустое, вернуть `400 Bad Request` с описанием ошибок по полям.

5. Backend выполняет проверку длины полей:
   - `firstName` — от 1 до 100 символов;
   - `lastName` — от 1 до 100 символов;
   - `email` — не более 255 символов;
   - `password` — от 8 до 64 символов;
   - `confirmPassword` — от 8 до 64 символов.
   - При нарушении ограничений вернуть `400 Bad Request`.

6. Backend валидирует формат email.
   - Если email имеет некорректный формат, вернуть `400 Bad Request`.

7. Backend сравнивает значения `password` и `confirmPassword`.
   - Если значения не совпадают, вернуть `400 Bad Request`.

8. Backend выполняет дополнительные проверки пароля по бизнес-правилам системы.
   Например:
   - наличие хотя бы одной буквы;
   - наличие хотя бы одной цифры;
   - отсутствие запрещенных символов, если это предусмотрено требованиями.
   - При нарушении вернуть `400 Bad Request`.

9. Backend нормализует email.
   Например:
   - удаляет лишние пробелы;
   - приводит значение к нижнему регистру.

10. Backend выполняет поиск пользователя в базе данных по email.
    - Если пользователь с таким email уже существует, вернуть `409 Conflict`.

11. Backend хеширует пароль.
    - Пароль в открытом виде в базу данных не сохраняется.

12. Backend формирует сущность пользователя для сохранения в базе данных.
    Обычно в сущность включаются:
    - уникальный идентификатор пользователя;
    - имя;
    - фамилия;
    - email;
    - хеш пароля;
    - дата создания;
    - статус пользователя.

13. Backend сохраняет нового пользователя в базе данных.
    - Если во время сохранения возникает техническая ошибка, вернуть `500 Internal Server Error`.

14. После успешного сохранения backend формирует успешный ответ.
    В ответе могут возвращаться:
    - `userId`
    - `firstName`
    - `lastName`
    - `email`
    - `status = SUCCESS`
    - сообщение об успешной регистрации

15. Backend возвращает ответ `201 Created`.

---

## Проверки, которые backend обязан выполнить до создания пользователя в БД

- Проверка корректности структуры JSON.
- Проверка наличия обязательных полей.
- Проверка на пустые значения.
- Проверка ограничений по длине полей.
- Проверка формата email.
- Проверка совпадения `password` и `confirmPassword`.
- Проверка соответствия пароля правилам безопасности.
- Проверка уникальности email в системе.
- Хеширование пароля перед сохранением.

---

## Возможные ответы сервиса

### Успешный сценарий
- `201 Created` — пользователь успешно создан.

### Ошибки на стороне клиента
- `400 Bad Request` — некорректный запрос, ошибка валидации, пустые или неверные поля.
- `409 Conflict` — пользователь с таким email уже существует.

### Ошибка на стороне сервера
- `500 Internal Server Error` — внутренняя ошибка backend-сервиса или базы данных.
